# StackTest Local Dashboard — Feature Design & AI Agent Implementation Plan

**Status:** Proposed  
**Target repo:** `https://github.com/gattasrikanth/stacktest`  
**Feature name:** Local Dashboard  
**Primary command:** `stacktest dashboard`  
**Secondary integration:** `stacktest run ... --dashboard`  
**Core principle:** Local-first, zero cloud dependency, ultra-low latency, safe by default.

---

## 1. Executive Summary

StackTest currently stores test run results in local gitignored folders under `.stacktest/runs/<run-id>`. This is already the perfect foundation for a local developer dashboard.

The goal of this feature is to provide a beautiful local web UI that allows users to inspect historical test runs, view per-resource deployment status, inspect CloudFormation event streams, open raw artifacts, and eventually watch active test runs live.

The first release should be read-only and safe. Later releases can add live run streaming and a guarded local test launcher.

The dashboard should make StackTest feel like a polished developer product instead of only a CLI tool.

---

## 2. Goals

### 2.1 Product Goals

1. Provide a local dashboard for all historical StackTest runs.
2. Make run debugging much easier than opening JSON/log files manually.
3. Make StackTest demo-friendly with a polished UI and screenshots.
4. Preserve the CLI-first workflow.
5. Keep everything local for now.
6. Create a clean architecture that can later evolve into a hosted/cloud dashboard.

### 2.2 Developer Experience Goals

Users should be able to run:

```bash
stacktest dashboard
```

and see:

```text
StackTest dashboard running at http://localhost:3456
Reading runs from .stacktest/runs
```

Users should also be able to run:

```bash
stacktest run examples/aws-cloudformation-basic/stacktest.yaml --dashboard
```

Expected behavior:

1. Run starts normally.
2. Local run artifacts are written as usual.
3. Dashboard opens automatically or prints the local URL.
4. Active run appears in the dashboard with live status.

---

## 3. Non-Goals for MVP

Do not build these in the first milestone:

1. Hosted SaaS dashboard.
2. Login/authentication.
3. Remote team sharing.
4. Persistent database requirement.
5. Full visual test authoring UI.
6. Cloud-based artifact upload.
7. Complex RBAC or enterprise admin controls.
8. Editing existing run artifacts from the UI.

These can come later after the local dashboard is stable.

## 4. Proposed UX

### 4.1 Main Navigation

The dashboard should have these top-level sections:

1. **Runs** — historical and active test runs.
2. **Run Detail** — selected run summary and deployment/test details.
3. **Artifacts** — raw files generated by the run.
4. **Settings** — local dashboard preferences.
5. **Launch Test** — future opt-in section, disabled by default in early release.

### 4.2 Run History Page

```text
/runs
```

Shows a table/card list of historical runs.

Columns:

| Field              | Description                           |
| ------------------ | ------------------------------------- |
| Run ID             | Example: `st-20260606-6e0feb`         |
| Scenario / Project | Example: `aws-basic-sandbox`          |
| Provider           | Example: `aws-cloudformation`         |
| Regions            | Example: `us-east-1`                  |

| Total Tests        | Count                                 |

Filters:

1. All
2. Passed
3. Failed
4. Running
5. Provider

7. Date range
8. Search by run ID, stack name, template path, scenario name

## 5. CLI Commands

### 5.1 Dashboard Command

```bash
stacktest dashboard
```

Options:

```bash
stacktest dashboard \
  --dir .stacktest \
  --port 3456 \
  --host 127.0.0.1 \
  --open \
  --mock false
```

Recommended options:

| Option             |           Default | Description                                        |
| ------------------ | ----------------: | -------------------------------------------------- |
| `--dir`            |      `.stacktest` | StackTest data directory                           |
| `--runs-dir`       | `.stacktest/runs` | Override run folder location                       |
| `--port`           |            `3456` | Local server port                                  |
| `--host`           |       `127.0.0.1` | Bind address. Must default to localhost only       |
| `--open`           |            `true` | Open browser automatically                         |
| `--no-open`        |           `false` | Do not open browser                                |
| `--mock`           |           `false` | Serve deterministic mock data for docs/screenshots |
| `--enable-actions` |           `false` | Allow dashboard to launch tests locally            |

Security requirement:

The default host must be `127.0.0.1`, not `0.0.0.0`.

### 5.2 Run Integration

```bash
stacktest run ./stacktest.yaml --dashboard
```

Behavior:

1. Start run.
2. Write run events to `.stacktest/runs/<run-id>/events.jsonl`.
3. Start dashboard server if not already running.
4. Open browser to `/runs/<run-id>`.

Also support:

```bash
stacktest run ./stacktest.yaml --dashboard --dashboard-port 3456
```

### 5.3 Future Run Launcher

Future command behind opt-in flag:

```bash
stacktest dashboard --enable-actions
```

Only then the browser UI may launch a run.

## 6. Artifact Model

### 6.1 Existing Assumption

Current runs are stored locally under:

```text
.stacktest/runs/<run-id>/
```

The dashboard must treat this folder as the source of truth.

### 6.2 Required New Files

Each run should eventually include these normalized files:

```text
.stacktest/runs/<run-id>/
  manifest.json
  summary.json
  events.jsonl
  deployments.json
  assertions.json
  logs/
  artifacts/
```

### 6.3 `manifest.json`

Purpose: stable entry point for the dashboard.

```json
{
  "schemaVersion": "1.0",
  "runId": "st-20260606-6e0feb",
  "scenarioName": "aws-basic-sandbox",
  "status": "PASSED",
  "startedAt": "2026-06-06T22:50:27.000Z",
  "endedAt": "2026-06-06T22:51:33.750Z",
  "durationMs": 66750,
  "provider": "aws-cloudformation",
  "regions": ["us-east-1"],
  "summaryPath": "summary.json",
  "eventsPath": "events.jsonl",
  "deploymentsPath": "deployments.json",
  "assertionsPath": "assertions.json"
}
```

### 6.4 `summary.json`

```json
{
  "runId": "st-20260606-6e0feb",
  "status": "PASSED",
  "totals": {
    "tests": 1,
    "passed": 1,
    "failed": 0,
    "skipped": 0
  },
  "durationMs": 66750,
  "deployments": [
    {
      "id": "sqs-queue",
      "name": "sqs-queue",
      "provider": "aws-cloudformation",
      "region": "us-east-1",
      "status": "CREATE_COMPLETE",
      "durationMs": 66750
    }
  ]
}
```

### 6.5 `events.jsonl`

Use append-only JSON Lines for low-latency live updates.

Each line is one event:

```json
{"schemaVersion":"1.0","eventId":"evt-001","runId":"st-20260606-6e0feb","timestamp":"2026-06-06T22:50:27.000Z","type":"run_started","message":"Run started"}
{"schemaVersion":"1.0","eventId":"evt-002","runId":"st-20260606-6e0feb","timestamp":"2026-06-06T22:50:29.000Z","type":"deployment_event","deploymentId":"sqs-queue","resourceLogicalId":"SandboxQueue","resourceType":"AWS::SQS::Queue","status":"CREATE_IN_PROGRESS","message":"Resource creation initiated"}
{"schemaVersion":"1.0","eventId":"evt-003","runId":"st-20260606-6e0feb","timestamp":"2026-06-06T22:51:32.000Z","type":"deployment_event","deploymentId":"sqs-queue","resourceLogicalId":"SandboxQueue","resourceType":"AWS::SQS::Queue","status":"CREATE_COMPLETE","message":"Resource creation complete"}
{"schemaVersion":"1.0","eventId":"evt-004","runId":"st-20260606-6e0feb","timestamp":"2026-06-06T22:51:33.750Z","type":"run_completed","status":"PASSED","message":"Run completed"}
```

### 6.6 Event Types

Recommended event types:

| Event Type           | Purpose                          |
| -------------------- | -------------------------------- |
| `run_started`        | New run started                  |
| `run_config_loaded`  | Config resolved                  |
| `deployment_started` | Deployment started               |
| `deployment_event`   | Provider-specific resource event |
| `assertion_started`  | Test assertion started           |
| `assertion_passed`   | Assertion passed                 |
| `assertion_failed`   | Assertion failed                 |
| `cleanup_started`    | Cleanup started                  |
| `cleanup_event`      | Cleanup resource event           |
| `cleanup_completed`  | Cleanup finished                 |
| `run_failed`         | Run failed                       |
| `run_completed`      | Run completed                    |
| `warning`            | Non-fatal warning                |
| `log`                | Generic log event                |

### 6.7 Backward Compatibility

The dashboard must not require all files to exist in old run folders.

Implement a tolerant reader:

1. Try `manifest.json`.
2. If missing, infer metadata from folder name and available files.
3. If only logs exist, show a minimal run card.
4. Mark incomplete folders as `UNKNOWN` instead of crashing.

## 7. Local Server Architecture

### 7.1 Recommended Stack

Backend:

1. Node.js
2. TypeScript
3. Fastify or Express
4. Chokidar for file watching
5. Server-Sent Events for live updates

Frontend:

1. React
2. Vite
3. TypeScript
4. Lightweight CSS/Tailwind or existing docs style tokens
5. No heavy state management needed for MVP

### 7.2 Package Structure

Recommended structure:

```text
packages/
  cli/
    src/commands/dashboard.ts
  core/
    src/runs/run-artifacts.ts
    src/runs/run-events.ts
  dashboard/
    package.json
    src/server/
    src/web/
    src/shared/
```

Alternative if repo currently avoids multiple packages:

```text
src/
  cli/commands/dashboard.ts
  dashboard/server/
  dashboard/web/
  dashboard/shared/
  runs/
```

Agent instruction:

Use the existing repo conventions. Do not force a new package structure if the current repository is simpler.

### 7.3 API Endpoints

Base URL:

```text
http://127.0.0.1:3456/api
```

Endpoints:

| Method | Route                                   | Purpose                                                 |
| ------ | --------------------------------------- | ------------------------------------------------------- |
| GET    | `/api/health`                           | Server health                                           |
| GET    | `/api/runs`                             | List runs                                               |
| GET    | `/api/runs/:runId`                      | Get run detail                                          |
| GET    | `/api/runs/:runId/events`               | Get all events                                          |
| GET    | `/api/runs/:runId/events/stream`        | SSE stream for live events                              |
| GET    | `/api/runs/:runId/artifacts`            | List artifact files                                     |
| GET    | `/api/runs/:runId/artifacts/file?path=` | Read artifact file                                      |
| GET    | `/api/settings`                         | Read local dashboard settings                           |
| PUT    | `/api/settings`                         | Update local dashboard settings                         |
| POST   | `/api/actions/run`                      | Future: launch test, disabled unless `--enable-actions` |
| POST   | `/api/actions/reveal`                   | Reveal run folder in OS file explorer                   |

### 7.4 SSE Event Stream

Route:

```text
GET /api/runs/:runId/events/stream
```

Use Server-Sent Events:

```text
event: stacktest-event
data: {"type":"deployment_event","status":"CREATE_IN_PROGRESS"}
```

Why SSE instead of WebSockets for MVP:

1. Simpler.
2. Browser-native.
3. One-way server-to-browser streaming is enough.
4. Lower implementation risk.
5. Easy to test.

WebSocket can be added later if dashboard actions require bidirectional realtime state.

### 7.5 File Watching

Use Chokidar to watch:

```text
.stacktest/runs/**/manifest.json
.stacktest/runs/**/summary.json
.stacktest/runs/**/events.jsonl
```

Rules:

1. Debounce file events by 50–100ms.
2. Avoid re-reading huge files fully on every change.
3. For `events.jsonl`, track last byte offset and only read appended content.
4. If a JSON line is partially written, buffer and retry after next file event.
5. Never crash the server due to malformed partial JSON.

## 9. Local Test Launcher — Future Phase

This should not be enabled by default.

### 9.1 Activation

Only available when server starts with:

```bash
stacktest dashboard --enable-actions
```

### 9.2 Required Safety Checks

Before launching a test from UI, show confirmation modal:

```text
This will deploy cloud resources.

Config: examples/aws-cloudformation-basic/stacktest.yaml
Provider: aws-cloudformation
Region: us-east-1
AWS Account: 123456789012
Cleanup: enabled

Continue?
```

User must click:

```text
Deploy and Run Test
```

### 9.3 Server Execution Model

Preferred approach:

1. Use internal StackTest runner API if available.
2. Avoid shelling out when possible.
3. If shelling out is necessary, use `child_process.spawn` with argument arrays only.
4. Never build shell command strings from user input.
5. Restrict file selection to current workspace unless explicitly configured.

Disabled by default.

```json
{
  "runId": "st-20260606-6e0feb",
```

## 13. Security and Safety

### 13.1 Local Binding

Default server binding:

```text
127.0.0.1
```

Do not bind to `0.0.0.0` by default.

### 13.2 File Access Safety

Artifact file APIs must:

1. Resolve paths safely.
2. Reject `../` path traversal.
3. Only read files under the selected run folder.
4. Avoid serving arbitrary local files.

### 13.3 Action Safety

Run-launching APIs must be disabled by default.

When enabled:

1. Require explicit `--enable-actions` flag.
2. Show UI warning before cloud deployment.
3. Avoid shell command injection.
4. Prefer internal APIs over shell strings.
5. Display resolved account and region before deploying.

### 13.4 Privacy

Docs and UI should clearly say:

```text
The StackTest dashboard is local-only. It reads local run artifacts from `.stacktest/runs`. No test data is uploaded by the dashboard.
```

---

## 14. Performance Requirements

1. Dashboard initial load should complete under 500ms for 100 runs on a normal laptop.
2. Run list should support at least 1,000 local run folders.
3. Use pagination or virtualized list if needed.
4. Event stream should append without re-rendering the entire page.
5. Large logs should be lazy-loaded.
6. JSON files over a size threshold should show preview with option to open raw.
7. File watching should debounce updates.
8. Avoid reading every artifact file during run list load.

## 17. Acceptance Criteria

### MVP Acceptance Criteria

1. `stacktest dashboard` starts a local web server.
2. Dashboard defaults to `127.0.0.1`.
3. Dashboard reads `.stacktest/runs`.
4. Historical runs appear in the UI.
5. User can open a run detail page.
6. User can see run summary metrics.
7. User can see deployment metadata.
8. User can see event timeline.
9. User can browse run artifacts.
10. Old/incomplete run folders do not crash the dashboard.
11. Docs include local dashboard usage.
12. Docs include at least three screenshots generated from mock data.
13. Tests pass.
14. Typecheck passes.
15. Lint passes.

### Live Mode Acceptance Criteria

1. Active run writes to `events.jsonl`.
2. Dashboard shows active run as running.
3. New events appear without browser refresh.
4. Run completion updates status and summary.
5. SSE disconnect/reconnect does not crash UI.

### Optional Launcher Acceptance Criteria

1. Launch UI is hidden unless `--enable-actions` is passed.
2. User sees confirmation before deployment.
3. Server does not execute shell strings built from user input.
4. Started run appears in dashboard.
5. Docs clearly mark launcher as local and opt-in.

## 20. Final Recommendation

Build this feature in this order:

1. Read-only local dashboard.
2. Live active run viewer.
3. Mock screenshots and docs updates.
4. Guarded local test launcher.

The read-only dashboard is the safest MVP and immediately valuable. Live streaming should come next because it makes StackTest feel real-time and premium. The UI test launcher is powerful, but it should be opt-in and carefully guarded because it can deploy cloud resources.
